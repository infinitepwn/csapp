
虽然可以使用低级mmap和munmap函数来创建和删除虚拟内存区域，但C程序员通常发现，当他们需要在运行时获取额外的[[虚拟内存]]时，使用动态内存分配器更方便、更便携。
动态内存分配器维护进程虚拟内存的一个区域，称为[[堆]]（图9.33）。细节因系统而异，不失去一般性，我们将假设堆是一个需求零的内核区域，在未初始化的数据区域之后立即开始，并向上增长（向更高的地址）。
![image.png|525](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260601164930198.png)

对于每个进程，内核都保持一个指向堆顶部的变量brk（发音为“break”）。分配器将堆维护为各种大小的块的集合。
每个块都是连续的虚拟内存块，该块要么被分配，要么被释放。分配的块已明确保留供应用程序使用。可以分配一个免费区块。在应用程序明确分配之前，自由块保持自由。分配的块一直被分配，直到它被释放，要么由应用程序显式分配，要麼由内存分配器本身隐式释放。

分配器有两种基本风格。这两种样式都要求应用程序明确分配块。对于哪个实体负责释放分配的块，他们有所不同。

## [[Explicit allocators]](显式分配器)
 需要应用显式地释放已经分配的块，C标准库里可以用malloc和free来分配释放块，在C++中对应的是new和delete
## [[Implicit allocators]](隐式分配器)
要求分配器检测分配的块何时不再被程序使用，然后释放该块。隐式分配器也被称为垃圾收集器，自动释放未使用的分配块的过程被称为垃圾收集。例如，Lisp、ML和Java等更高级别的语言依赖于垃圾收集来释放分配的块

## 为什么需要动态内存分配
最重要原因是，在程序实际运行之前，通常不知道某些数据结构的大小。
例如，假设编写将n个ASCII整数列表（每行一个整数）从stdin读取到C数组中的程序。输入由整数n组成，然后是n個要读取和存储到数组中的整数。
最简单的方法是用一些硬编码的最大数组大小静态定义数组：

```C
#include "csapp.h"
#define MAXN 15213
int array[MAXN];
int main()
{
	int i, n;
	scanf("%d", &n);
	if (n > MAXN)
		app_error("Input file too big");
	for (i = 0; i < n; i++)
		scanf("%d", &array[i]);
	exit(0);

}
```

像这样用硬编码大小分配数组通常是一个坏主意。MAXN的值是任意的，与机器上的实际可用虚拟内存量无关。此外，如果该程序的用户想要读取比MAXN更大的文件，唯一的手段是重新编译具有更大MAXN值的程序。虽然对于这个简单的例子来说不是问题，但对于拥有数百万行代码和众多用户的大型软件产品来说，硬编码数组边界的存在可能会成为维护噩梦。更好的方法是，在知道n的值后，在运行时动态分配数组。使用这种方法，阵列的最大大小仅受可用虚拟内存量的限制。

## 分配器要求

显式分配器必须在一些相当严格的约束范围内操作：
### 处理任意请求序列
应用程序可以任意执行分配和释放请求的数列，但受制，每个自由请求必须对应于从之前的分配请求中获得的当前分配块。因此，分配器不能对分配和自由请求的排序做出任何假设。例如，分配器不能假设所有分配请求都伴随着一个匹配的自由请求，或者匹配的分配和自由请求是嵌套的。
### 立即响应请求
分配者必须立即响应以分配请求。因此，分配器不允许为了提高性能而重新排序或缓冲请求。
### 仅使用堆
为了使分配器具有可扩展性，分配器使用的任何非標量数据结构都必须存储在堆本身中。
### 对齐块（对齐要求）
分配器必须以能够容纳任何类型的数据对象的方式对齐块。

### 最大化吞吐量

给定n个分配和自由请求
R0, R1, . . . , Rk, . . . , Rn−1
的某些序列，我们想最大化分配器的吞吐量，它被定义为它每单位时间内完成的请求数量。
$$吞吐量 = \frac{malloc请求 + free请求}{总时间}$$
例如，如果分配器在1秒内完成500个分配请求和500个自由请求，那么其吞吐量为每秒1000次操作。
一般來說，我們可以透過最小化滿足分配和自由請求的平均時間來最大化吞併。正如我们将看到的，开发具有相当良好性能的分配器并不难，malloc请求的最坏运行时间在自由块数量上是线性的，free请求的运行时间是恒定的。

## 碎片

### [[内碎片]]
当分配的块大于payload时，就会发生内部碎片化。发生这种情况的原因有很多。
例如，分配器的实现可能会对分配的块施加大于某些请求的有效负载的最小大小。
或者，正如我们在图9.34（b）中看到的，分配器可能会增加块大小，以满足对齐约束。

### [[外碎片]]
当有足够的聚合可用内存来满足分配请求，但没有单个可用块足够大来处理请求时，就会发生外部碎片化。
例如，如果图9.34（e）中的请求是八个单词而不是两个单词，那么如果不从内核请求额外的虚拟内存，即使堆中还剩下八个空闲单词，也无法满足该请求。出现这个问题是因为这八个单词分布在两个自由块上。
