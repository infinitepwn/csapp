本讲概览

  • 并行计算硬件
    • 多核（Multicore）：单芯片上多个独立处理器
    • 超线程（Hyperthreading）：单核高效执行多线程
  • 线程级并行
    • 将程序划分为独立任务
    • 示例1：并行求和
    • 示例2：并行快速排序（分治并行）
  • 一致性模型：多线程读写共享状态时会发生什么
二

并行执行的动机

  • 之前用线程处理 I/O 延迟（如每个客户端一个线程，防止互相阻塞）
  • 多核/超线程 CPU 提供了新机会：
    • 将工作分散到并行执行的线程上
    • 多个独立任务时自动并行（如运行多个程序、服务多个客户端）
    • 也可以将一个大任务组织成多个并行子任务，从而加速
三

硬件基础

3.1 典型多核处理器结构

 Core 0                     Core n-1
┌──────────────────┐        ┌──────────────────┐
│  Regs            │  ...   │  Regs            │
│  L1 d-cache      │        │  L1 d-cache      │
│  L1 i-cache      │        │  L1 i-cache      │
│  L2 unified cache│        │  L2 unified cache│
└──────────────────┘        └──────────────────┘
         ┌──────────────────────────────┐
         │   L3 unified cache（共享）    │
         └──────────────────────────────┘
                     Main Memory

多个处理器对内存有一致的视图（Coherent view of memory）。

3.2 乱序执行（Out-of-Order）处理器

  • 指令控制单元动态地将程序转换为操作流
  • 操作映射到多个功能单元（FU）上并行执行：整数算术单元（×2）、浮点单元、Load/Store 单元

3.3 超线程实现（Hyperthreading）

  • 复制足够的指令控制来处理 K 个指令流
  • K 份寄存器拷贝（每个线程独立的 PC、寄存器堆、操作队列）
  • 共享功能单元
  • 原理：当一个线程等待（如 cache miss）时，另一个线程利用空闲的功能单元

3.4 基准测试机器

  • Intel Xeon E5520 @ 2.27 GHz（Nehalem 架构，约2010年）
  • 8个核，每核支持 2× 超线程
  • 信息来源：/proc/cpuinfo

示例1：并行求和

任务：求 0 到 n-1 的总和（结果应为 (n-1)*n/2）

策略：将值域 [1, n-1] 分成 t 个范围，每个线程处理一个范围（假设 n 是 t 的倍数）。

4.1 方法一：全局变量（有竞态问题）

C/* 错误示范：直接累加全局变量 */
void *sum_global(void *vargp) {
    // ... 直接累加到全局变量 global_sum
    // ❌ 问题：存在竞态条件（race condition），结果不可预期
}
⚠️
多线程同时写全局变量，无互斥保护时会发生竞态，结果不正确且不可复现。

4.2 方法二：局部变量 + 全局数组（正确）

C/* 每个线程先算局部和，再写入全局数组各自的槽 */
void *sum_local(void *vargp) {
    long myid  = ((arg_t *)vargp)->id;
    long start = myid * (n / nthreads);
    long end   = start + n / nthreads;
    long sum   = 0;
    for (long i = start; i < end; i++)
        sum += i;
    gsum[myid] = sum;  // ✅ 每个线程只写各自的槽
    return NULL;
}

主线程最后汇总 gsum[] 中各线程的局部和。

4.3 性能分析：Amdahl's Law

线程数理想加速比实际加速比
11×~1×
22×~1.99×
44×~3.91×
88×~7.08×
1616×~13.2×
💡
超过物理核数（>8）后，加速比增长放缓，受超线程效率和串行部分的限制。

4.4 伪共享（False Sharing）

即使每个线程写不同的数组元素，若这些元素在同一 cache line 内，写一个槽会使其他核缓存中同一 cache line 失效（invalidate），引发不必要的缓存同步，严重降低性能。

解决方案：对数组元素加 padding，保证每个线程的数据落在不同的 cache line 上。

C// 每个槽 padding 到 64 字节（一个 cache line）
typedef struct {
    long val;
    long pad[7];  // 7 × 8 + 8 = 64 字节
} padded_t;
padded_t gsum[MAXTHREADS];

示例2：并行快速排序

5.1 顺序快速排序

Cvoid qsort_serial(data_t *base, size_t nele) {
    if (nele <= 1) return;
    if (nele == 2) {
        if (base[0] > base[1]) swap(base, base+1);
        return;
    }
    size_t m = partition(base, nele);  // 返回枢轴索引
    if (m > 1)        qsort_serial(base, m);
    if (nele-1 > m+1) qsort_serial(base+m+1, nele-m-1);
}

过程：选择枢轴 p → 重排为 L（≤p）、p、R（≥p）→ 递归排序 L 和 R → 返回 L : p : R

5.2 并行快速排序思路

  • 若子数组大小 N ≤ Nthresh，退化为顺序快速排序
  • 否则：选枢轴 → 划分 L / R → 递归地 spawn 独立线程分别排序 → 合并

5.3 顶层函数（简化）

Cvoid tqsort(data_t *base, size_t nele) {
    init_task(nele);
    global_base = base;
    global_end  = global_base + nele - 1;
    task_queue_ptr tq = new_task_queue();
    tqsort_helper(base, nele, tq);
    join_tasks(tq);        // 等待所有子线程完成
    free_task_queue(tq);
}

5.4 递归排序函数（简化）

Cstatic void tqsort_helper(data_t *base, size_t nele,
                           task_queue_ptr tq) {
    if (nele <= nele_max_sort_serial) {
        qsort_serial(base, nele);  // 小数组用顺序排序
        return;
    }
    size_t m = partition(base, nele);
    spawn_task(tq, tqsort_helper, base,     m,          tq);
    spawn_task(tq, tqsort_helper, base+m+1, nele-m-1, tq);
}

5.5 并行划分（Parallel Partitioning）的尝试

❌
实验结论：并行化 partition 步骤无法获得加速比。
推测原因：数据拷贝开销过大——需要临时空间重新组装划分结果，无法在原地完成。
六

经验总结与教训

  1. 必须有并行化策略：将问题分成 K 个独立部分（分区法）或使用分治（divide-and-conquer）
  2. 内层循环必须无同步：同步操作（锁、原子操作等）代价极高，会严重拖慢性能
  3. 注意 Amdahl's Law：串行代码会成为瓶颈，极小比例的串行部分也能严重限制加速比上限
    📐
    Amdahl's Law 公式：  S = 1 / (s + (1-s)/p)  ，其中 s 为串行比例，p 为处理器数量
  4. 实践是可行的：实现适度并行加速并不难，搭建实验框架，测试多种策略。
七

内存一致性（Memory Consistency）

7.1 问题引入

Cint a = 1, b = 100;

// Thread 1:               // Thread 2:
a = 2;       /* Wa */    b = 200;     /* Wb */
print(b);    /* Rb */    print(a);    /* Ra */

可能的输出值是什么？—— 这取决于内存一致性模型（Abstract model of how hardware handles concurrent accesses）。

7.2 顺序一致性（Sequential Consistency）

整体执行效果与某种所有线程操作的顺序交织（interleaving）一致，且每个线程内的操作顺序保持不变。

执行顺序Thread1 输出（Rb）Thread2 输出（Ra）
Wa→Rb→Wb→Ra1002
Wa→Wb→Rb→Ra2002
Wb→Ra→Wa→Rb1200
Wb→Wa→Ra→Rb2200
🚫
不可能的输出：100, 1 或 1, 100——这需要在 Wa 和 Wb 都执行前就读到 Ra 和 Rb，违反线程内顺序约束。

7.3 非一致缓存场景（Non-Coherent Cache）

  • 写回缓存（write-back cache）但缓存之间没有协调
  • 结果：Thread1 的 a=2 存在 Thread1 的缓存中，但 Thread2 读 a 时仍从主存读到 a=1
  • 这会导致 print(a)=1, print(b)=100 这种在顺序一致性下不可能出现的结果

7.4 Snoopy 缓存（缓存一致性协议）

每个缓存块打上状态标签（MSI 协议）：

Invalid

不可使用该值（已失效），需要从其他缓存或主存重新获取

Shared

可读的拷贝，可能多个缓存共同持有，但不能直接写

Exclusive

可写的拷贝，只有一个缓存持有，值最新

工作机制：当某缓存看到其他缓存对自己 Exclusive 标记的块发出请求时，从本地缓存（而非主存）提供最新值，并将状态降级为 Shared。

✅
效果：保证所有核看到的是内存的一致视图，防止读到旧值，实现顺序一致性。
