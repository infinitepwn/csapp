## 代码移动(code motion)

我们可以把循环里一些要执行多次但是计算结果不会改变的计算移动到不会被多次执行的部分
``` C
void combine1(vec_ptr v, data_t *dest)
{
    long i;
    *dest = IDENT;
    for (i = 0; i < vec_length(v); i++){
        data_t val;
        get_vec_element(v, i, &val);
        *dest = *dest OP val; 
    }
}
void combine2(vec_ptr v, data_t *dest)
{
    long i;
    long length = vec_length(v);
    *dest = IDENT;
    for (i = 0; i < length; i++){
        data_t val;
        get_vec_element(v, i, &val);
        *dest = *dest OP val; 
    }
}
```
可以看到有一定提升，但是当n越来越大，这种提升会越来越小
![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260627161449749.png)

不过不能小瞧这种改动，可能会有隐藏的渐进低效率
比如lower1和lower2
一个是O(n)一个是O(n^2)

## 用 CPE 看 combine1 和 combine2

`combine1` 的循环条件是 `i < vec_length(v)`，所以每次迭代都要重新调用 `vec_length`：

```C
for (i = 0; i < vec_length(v); i++)
```

`combine2` 把长度提前保存：

```C
long length = vec_length(v);
for (i = 0; i < length; i++)
```

这属于代码移动。被移动的表达式在循环过程中不会改变，因此没有必要每次迭代都重新计算。

用 `/opt/measure_combine_cpe.py` 对不同 `n` 测量后，可以看到小规模时固定开销占比更高；`n` 增大后，每个元素的平均开销趋于稳定。本机一次测量结果：

```text
n=1048576
combine1: 0.250 ns/element
combine2: 0.370 ns/element
combine3: 0.068 ns/element
```

这里 `combine1` 和 `combine2` 的相对顺序会受编译优化、内联、机器频率调度影响，不一定每次都严格 `combine2 < combine1`。但是核心结论不变：把循环不变量移出循环可以减少循环体工作量；如果这个不变量本身代价很高，或者会改变算法复杂度，收益会非常明显。
